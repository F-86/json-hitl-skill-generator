## 修改方向分类

### 方向 1：查询流程架构演进（CP-1a/CP-1b 的取舍）

**经历的三种方案：**

1. **方案 A**（最初）：CP-1a 收集 → LLM 生成 apicall → 执行
   - 问题：LLM 跨轮生成 apicall 时容易丢失 filter 参数
2. **方案 B**（中间）：CP-1a 收集 → CP-1b 确认（readonly 展示 + choice + 内嵌 apicall）→ 前端执行
   - 问题：用户被确认两次（CP-1a 填表 + CP-1b 再确认），体验冗余
3. **方案 C**（最终）：CP-1a 收集（checkpoint 内嵌 apicall 模板）→ 前端提交时直接注入 filters 并执行
   - 优势：只确认一次，LLM 不参与提交环节，不会丢参数

**核心教训：**
- 查询场景不需要"最终确认"步骤（与 Create/Update/Delete 不同）
- CP-1a 的 checkpoint 应内嵌 `apicall` 模板（`filters: {}`），前端提交时填充
- LLM 不应参与表单提交后的环节——前端直接执行，避免跨轮丢参数

### 方向 2：参数提取与预填策略

**问题：**
- LLM 对"今年"等时间表达式理解错误（把 2026 当 2025）
- LLM 对"今年"的结束时间取了今天而非年底
- LLM 有时不用 default 预填已提取的参数
- LLM 有时分多轮询问（先问价格、再问分类），而非一次性收集

**解决：**
- system prompt 注入当前日期
- SKILL.md 明确时间表达式解析规则表（今年→01-01~12-31）
- 强制"CP-1a 一轮查询只出一次"，禁止分多轮询问
- 已提取的参数用 `default` 字段预填到 decision

### 方向 3：decision 类型扩展

**新增 `text_input` 类型：**
- 用于 id 和 name 等需要自由文本输入的查询字段
- 支持 `default` 预填、`placeholder` 提示
- 前端 `buildFilters` 对 id（逗号分隔→number[]）和 name（逗号分隔→string/string[]）做格式转换
- 明确"中英文逗号均可"的 placeholder

**之前的问题：**
- 模板中禁止查询场景用 `input` 类型，但 id/name 没有合适的 decision 类型
- 最初把 id/name 排除在 CP-1a 之外，导致前端提交时覆盖了 LLM 预填的 filters

### 方向 4：后端鲁棒性

**max_tokens 截断：**
- 512 token 不够输出完整 hitl JSON，改为 2048

**裸 JSON 兜底：**
- LLM 有时不加 ```hitl 包裹直接输出裸 JSON
- 后端增加兜底：检测到 `{` 开头且含 `"checkpoint"` 的回复，直接解析为 hitl

**多值字段支持：**
- name 字段从 string 扩展为 string|string[]，后端用 OR 拼接多个 LIKE

### 方向 5：前端交互修复

**IME 回车误触发：**
- 中文输入法下按回车确认选字，不应触发消息发送
- `handleKeyDown` 增加 `!e.nativeEvent.isComposing` 判断

**HITL 提交时 filters 合并：**
- 前端 `handleSubmit` 构造 `filledApicall` 时，要用 `{ ...baseFilters, ...out }` 合并 LLM 预填的 filters（id/name）和表单填写的字段
- 否则表单值会覆盖 LLM 预填的条件

### 方向 6：LLM 行为约束

**HITL 块必须用代码块包裹：**
- 禁止裸 JSON 输出，否则前端无法渲染

**路由识别规则：**
- 明确"查全部"才跳过 CP-1a，其他一律走 CP-1a
- 禁止"没有收到表单提交就直接执行查询"

**CP-1a 只出一次：**
- 禁止"先提取部分参数 → 出 CP-1a → 再出 CP-1a 询问其他条件"
- 所有参数在同一个 CP-1a 中一次性收集

## 与 json-hitl-skill-generator 的关联

上述修改方向暴露了生成器在以下方面的不足（详见 `2026-06-24-query-improvements.md`）：

1. 查询 skill 的工作流模板（无最终确认 + apicall 内嵌）
2. `text_input` decision 类型
3. 时间表达式解析规则
4. LLM 行为约束（裸 JSON、CP 只出一次、default 预填）
5. 后端消费指南（max_tokens、日期注入、裸 JSON 兜底）
6. 前端消费指南（IME、filters 合并）
7. 参数规范表格的"收集方式"列
