## 二、本次对话里实际发生的修改方向

### 方向 1：把 Delete 从“单条确认”升级为“单条 + 按条件批量删除”双路径

原来删除 skill 更偏单条删除，本次把它扩成两条路线：

- **单条路径**：按 `id` 删除，最终确认后执行 `DELETE /api/products/{id}`
- **批量路径**：按 `filters` 删除，先 `dry_run` 预览，再最终确认，正式执行 `POST /api/products/bulk_delete`

关键经验：

- Delete 的批量路径和 Update 的批量路径应尽量**对称化设计**
- 不能把 Delete 永远视作“只适合单条按 ID 执行”
- 生成器应能产出“单条删除 + 批量删除”的双分支模板，而不是只会生成单条删除确认

---

### 方向 2：批量 Delete 也必须走 `dry_run → 预览确认 → 最终确认/执行` 三段流

这次落地出的批量删除链路为：

1. **Step 1**：输出 `dry_run=true` 的 `bulk_delete` apicall
2. **CP-2a**：前端展示命中列表后，进入“预览确认”
3. **CP-2b**：最终确认时再执行正式删除 apicall

关键经验：

- 不是只有批量 Update 需要 `dry_run`
- 批量 Delete 同样是高风险操作，也必须先让用户看到**命中范围**
- 生成器当前若只把 `dry_run` 视作 Update 专属，就会低估 Delete 的风险控制需求

---

### 方向 3：批量 Delete 的正式执行也要带 `expected_count` 双保险

本次补的后端接口支持：

- `dry_run=true` 返回 `matched` + `items`
- `dry_run=false` 时接收 `expected_count`
- 若正式执行时实际命中数与 `expected_count` 不一致，则 409 阻断

关键经验：

- `expected_count` 不应只属于批量 Update
- 凡是“预览后再正式执行”的批量写操作，都应该支持 `expected_count`
- 生成器需要把 `expected_count` 视作**通用批量写操作护栏**，而不是 Update 特例

---

### 方向 4：前端可能需要把“继续确认”归一化为结构化 HITL 回执

为把 `dry_run.matched` 带给下一步，本次前端不是只发送纯字符串 `approve`，而是会在合适场景回传：

```json
{"action":"approve","expected_count":3}
```

关键经验：

- 某些多步 HITL 流程，第二步需要复用前一步 apicall 的结果
- 这时消费侧需要一个“**回执归一化**”机制，把按钮点击转成结构化 JSON 回执
- 生成器当前虽然强调输出块协议，但对“用户回执如何结构化回传给 LLM/skill”写得还不够

---

### 方向 5：运行时 JSON fence 和文档示例 fence 必须严格区分

本次踩到了一个很实际的问题：

- 运行时 LLM 在第二轮把 checkpoint JSON 包进了 ```text 代码块
- 后端原先只认 ```hitl，结果前端没渲染出 checkpoint

后来采取了两种修复：

- **消费侧兼容**：后端兼容解析 ```text 中包裹的 JSON
- **skill 侧收紧**：明确要求运行时 checkpoint JSON 必须使用 ```hitl，而不是 ```text

关键经验：

- `json-hitl-skill-generator` 里已经有“文档示例块用 ```text，避免嵌套 fence 冲突”的规则
- 但还需要更强地强调：**示例展示**用 `text`，**实际运行输出**必须是 `hitl` / `apicall`
- 否则生成的 skill 很容易把“示例写法”误当作“运行时写法”

---

### 方向 6：消费侧不应把自然语言用户回复折叠成 AI 的 HITL 附属回执

本次还踩到了一个 UI/交互语义问题：

- 前端曾把“上一条 AI HITL 后的下一条 user 消息”统一视为回执，并折叠进 AI 卡片
- 结果像“把可乐删了”这种**自然语言**用户输入，看起来像“消失了”

最终修正后的规则是：

- **内部回执**（如 `approve` / `confirm` / JSON）可折叠或内嵌
- **自然语言消息**必须保留为独立用户气泡

关键经验：

- 在 HITL consumer 里，“用户自然语言”与“内部协议回执”必须区别对待
- 生成器除了定义输出协议，还需要提醒消费侧：**不要把自然语言用户消息语义降格为控件回执**

---

### 方向 7：前端不仅要渲染 HITL/apicall，还要渲染批量删除的预览与执行结果

为了让 Delete 的三段流可用，本次前端新增了：

- `bulk_delete` dry_run 预览结果展示
- `bulk_delete` 正式删除结果展示
- 预览界面的 matched/items 信息展示

关键经验：

- 生成器不能只说“前端会自动执行 apicall”
- 对于批量写操作，前端还需要有**预览态**和**执行完成态**两类结果卡片
- 否则 skill 虽然协议正确，但用户体验和风控闭环是不完整的

---

## 三、从这次会话抽象出来的 4 个核心教训

### 1. 批量写操作的风险控制模式应该统一抽象

目前已经验证成立的统一模式是：

- `dry_run`
- 预览确认
- 结构化回执（可选）
- 正式执行
- `expected_count` 双保险

这套模式不应只服务于 Update，也应覆盖 Delete，未来也可能扩展到其他批量写操作。

### 2. 生成器除了“产出 SKILL.md”，还必须指导消费侧如何正确消费

这次暴露的问题里，有一半不是 skill 文本本身有错，而是**消费侧契约没被写清**：

- 如何从前置 apicall 结果构造结构化回执
- 如何兼容 `text` fence 误包裹
- 如何区分内部回执和自然语言用户消息
- 如何渲染批量预览 / 最终结果

### 3. 运行时协议和文档示例语法必须分层说明

“文档示例为了避免嵌套 fence，用 `text` 展示”是文档约束；
“真实运行时只能输出 `hitl` / `apicall`”是协议约束。

两者若不分层写清，生成器就会把读者和运行时弄混。

### 4. Delete 不能再被当作 CRUD 中最简单的操作

单条 Delete 看起来简单，但一旦进入：

- 条件删除
- 预览后执行
- 多轮确认
- 前后端协作消费

它和批量 Update 一样，已经是一个跨层流程设计问题，而不是单个确认框问题。
